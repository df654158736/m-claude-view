# 文档同步（Doc-Sync）

## 核心原则

> **代码是 Source of Truth** — 需求文档、详设文档必须反映当前实现状态。
> **正文只保留当前态** — 历史变更全部外置到 `.changelog.md`。
> **双向同步** — 改需求→同步详设；改实现→同步需求。任何一方偏离就更新所有关联文档。

---

## 两种模式

| | 增量同步 | 全量同步 |
|--|--|--|
| **命令** | `/doc-sync` | `/doc-sync-after-dev` |
| **时机** | 开发过程中，发现实现偏离文档时 | feature 分支完成，准备合并前 |
| **触发方式** | 对话中识别到变更决策 / 手动调用 | 手动调用 / CLAUDE.md 规则要求 |
| **对比基准** | 当前对话中的变更决策 | `git diff master...HEAD` 全量 |
| **更新范围** | 只更新偏离的具体章节 | 扫描所有关联文档 |
| **版本号** | 递增（如 1.0→1.1） | 递增（如 1.1→1.2） |
| **执行步骤** | Step 1→2→3→4 | Step 1→2→3→4→5→6 |

---

## 文档格式统一规则（需求文档和详设文档通用）

### 规则 1：正文 + changelog 分离

每个文档拆成两个文件：

```
PRD-001-Tool-Policy-Engine.md              ← 正文，只保留当前最新状态
PRD-001-Tool-Policy-Engine.changelog.md    ← 变更日志，所有留痕集中在这里
```

详设同理：

```
SPEC-001-tool-policy-engine.md             ← 正文
SPEC-001-tool-policy-engine.changelog.md   ← 变更日志
```

### 规则 2：正文头部格式

```markdown
> **Version**: 1.2
> **Date**: 2026-04-02
> **Changelog**: [变更日志](./PRD-001-Tool-Policy-Engine.changelog.md)
```

其余已有头部字段（Author、Status、关联需求等）保持不变。

### 规则 3：正文章节变更标记

被变更过的章节，标题后加版本标记，标题下加一行变更摘要：

**变更 1-3 次**：

```markdown
### 3.2 分页导出 `v1.2`
> 变更: 废弃一次性导出，改为分页 `v1.2` 丁昂 | [详情](./PRD-001.changelog.md#v12)

WHEN 用户点击导出 AND 数据量 > 1000 THEN 分页生成 Excel...
```

**变更 >3 次**：只保留最近 3 次 + 总数

```markdown
### 2.1 工具发现策略 `v1.5`
> 变更: ...共 5 次 | 增加缓存 `v1.5` 丁昂 | 超时改为动态 `v1.4` 丁昂 | 默认超时 30→15s `v1.3` 丁昂 | [全部详情](./PRD-001.changelog.md#§21)
```

**从未变更**：不加任何标记。

### 规则 4：changelog 文件格式

```markdown
# {文档编号} 变更日志

> 关联正文: {正文文件名}
> 关联需求/详设: {关联文档路径}（按方向填写）

## [v1.2] 2026-04-02 | 丁昂

- **§3.2** 废弃 → 一次性导出改为分页导出 | 原因: 大数据量超时 | 替代: §3.2 分页导出 | 关联: SPEC-001 §4.1 同步更新
- **§5.1** 新增 → 批量操作 API | 原因: 开发中发现单条效率低 | 关联: SPEC-001 §5.1

## [v1.1] 2026-03-28 | 丁昂

- **§3.1** 修改 → 默认超时从 30s 改为 15s | 原因: 性能测试发现 30s 太慢 | 关联: PRD-001 §2.3 验收标准更新

## [v1.0] 2026-03-24 | 丁昂

- 初始创建
```

changelog 条目格式：

**简单变更**（仅影响小局部，一行即可说清）：
```markdown
- **§11** 修改 → 优先级矩阵更新 | 关联: §3.5, §3.7 优先级调整
```

**实质变更**（涉及功能逻辑、验收标准、接口定义等，必须保留原内容）：
```markdown
- **§3.7** 修改 → 工具策略优先级重构
  - 原内容: FR-3.7.1~3.7.7 全部标记为 P2，包含创建场景、执行 Action、差异高亮...
  - 现内容: P0 基础策略（创建+执行+差异高亮）、P1 规则变更、P2 多场景对比
  - 原因: 基础策略是核心能力，不应放到第二期
```

changelog 条目字段：

| 字段 | 说明 | 必填 |
|------|------|------|
| 章节号 | `§X.X` 对应正文章节 | 是 |
| 变更类型 | 新增 / 修改 / 废弃 / 推迟 | 是 |
| 变更内容 | 具体改了什么（一行摘要） | 是 |
| 原内容 | 变更前的内容描述（正文已删除，这里是唯一留存） | 修改/废弃时必填 |
| 现内容 | 变更后的内容描述 | 修改时必填 |
| 原因 | 为什么改 | 是 |
| 替代 | 废弃时填，指向替代方案的章节 | 废弃时必填 |
| 关联 | 同步更新了哪些其他文档的哪个章节 | 有关联时必填 |

> **关键原则**：正文只保留当前态，changelog 是原内容的唯一留存处。修改和废弃的条目必须写清"原来是什么"，否则变更历史不可追溯。

### 规则 5：废弃逻辑处理

- **正文**：直接删除废弃内容，替换为新方案。不留删除线。
- **changelog**：记录完整的废弃信息（原内容摘要、原因、替代方案）。
- **正文章节标记**：变更类型写"废弃"即可。

示例 — 正文：
```markdown
### 3.2 分页导出 `v1.2`
> 变更: 废弃一次性导出，改为分页 `v1.2` 丁昂 | [详情](./PRD-001.changelog.md#v12)

WHEN 用户点击导出 AND 数据量 > 1000 THEN 分页生成 Excel...
```

示例 — changelog：
```markdown
- **§3.2** 废弃 → 一次性导出改为分页导出 | 原因: 大数据量超时 | 原内容: "WHEN 用户点击导出 THEN 一次性生成完整 Excel" | 替代: §3.2 分页导出 | 关联: SPEC-001 §4.1
```

### 规则 6：版本号规则

| 变更类型 | 版本递增 | 示例 |
|---------|---------|------|
| 接口/数据模型/验收标准变更 | 次版本 +1 | 1.0 → 1.1 |
| 仅描述修正/格式调整 | 修订版本 +1 | 1.0.0 → 1.0.1 |
| 架构重大变更/需求大幅调整 | 主版本 +1 | 1.0 → 2.0 |

### 规则 7：现有变更记录迁移

首次对一个文档执行 `/doc-sync` 或 `/doc-sync-after-dev` 时：

1. 检查文档内是否有 `## 变更记录` / `## 变更日志` / `## Changelog` 表格
2. 如果有 → 将表格内容迁移到新建的 `.changelog.md`，从正文中删除该表格
3. 在正文头部补上 `Changelog` 指针
4. 迁移后的 changelog 条目格式尽量对齐规则 4（可能缺少章节号和原因，标注 `[迁移自旧格式]`）

迁移示例 — 从：
```markdown
## 变更记录

| 版本 | 日期 | 作者 | 变更 |
|------|------|------|------|
| 1.0 | 2026-03-27 | Claude | 初始版本 |
| 1.1 | 2026-04-01 | 丁昂 | Review 修复：Tool Policy 三步流程... |
```

迁移到 `.changelog.md`：
```markdown
## [v1.1] 2026-04-01 | 丁昂

- [迁移自旧格式] Review 修复：Tool Policy 三步流程完整设计(parse+execute)、ToolConfig 扩展为 4 类选择...

## [v1.0] 2026-03-27 | Claude

- 初始创建
```

## 增量同步模式（`/doc-sync`）

### 触发流程

```
Claude 完成代码实现
    ↓
自查：本轮变更是否偏离需求文档或详设文档？
    ↓ 偏离
告知用户偏离点，询问："要现在同步文档，还是稍后统一更新？"
    ↓ 现在更新              ↓ 稍后统一
    执行 /doc-sync         记住待同步项，继续开发
                                ↓
                    用户说"功能做完了"/"可以了"/"没其他改的了"
                                ↓
                    Claude 主动提醒："本轮有 N 处文档偏离待同步：
                    {清单}，现在统一更新文档吗？"
                                ↓ 用户确认
                            执行 /doc-sync（合并所有待同步项）
```

### 触发场景

| 场景 | 说明 |
|------|------|
| 用户做了决策且 Claude 完成实现 | "改字段"、"加功能"、"不做了"、"换方案" → 实现完 → 询问同步时机 |
| Claude 发现代码与文档不一致 | 实现过程中发现偏离 → 告知用户 → 询问同步时机 |
| 累积了待同步项且用户表示开发完成 | Claude 主动提醒统一更新 |
| 用户主动说"同步文档" | 手动调用 `/doc-sync`，跳过询问直接执行 |

### 不触发场景

| 场景 | 原因 |
|------|------|
| 纯样式/格式/typo 修改 | 不涉及功能逻辑 |
| 代码与文档一致 | 无偏离则不同步 |

### Step 1: 识别变更

从当前对话上下文提取已确认的变更决策：

**提取内容**：
1. **变更了什么**：哪个功能/接口/字段/行为发生了变化
2. **原来是什么**：文档中原来的描述
3. **现在是什么**：当前确认的实现方案
4. **为什么变**：变更原因
5. **谁发起的**：从 `git config user.name` 获取

**输出**：变更决策清单
```
[变更-1] §3.2 导出方式: 一次性导出 → 分页导出 | 原因: 大数据量超时 | 决策人: 丁昂
[变更-2] §5.1 批量操作: 无 → 新增批量 API | 原因: 单条效率低 | 决策人: 丁昂
```

### Step 2: 定位关联文档

根据变更内容定位所有需要更新的文档：

**搜索链路**：
1. 从当前工作的文件/模块 → 找到关联的详设文档（`docs/specs/`）
2. 从详设文档头部 `关联需求` 字段 → 找到需求文档（`sprints/backlog/requirements/`）
3. 反向：从需求文档 → 找到所有引用该需求的详设文档

**输出**：关联文档清单
```
需求文档: sprints/backlog/requirements/PRD-001-Tool-Policy-Engine.md
详设文档:
  - docs/specs/SPEC-001-tool-policy-engine.md
```

### Step 3: 更新文档

对每个关联文档执行：

#### 3.1 检查是否需要迁移旧格式

按规则 7 检查并迁移现有变更记录。

#### 3.2 递增版本号

按规则 6 递增正文头部的 Version 和 Date。

#### 3.3 更新正文内容

- **新增的功能**：在对应章节补充描述
- **修改的内容**：直接更新正文为当前最新状态
- **废弃的逻辑**：从正文删除，替换为新方案（如有）

#### 3.4 添加正文章节标记

按规则 3 在变更章节标题后加版本标记和变更摘要行。

#### 3.5 追加 changelog 条目

按规则 4 在 `.changelog.md` 中追加新版本条目。如果 changelog 文件不存在则创建。

### Step 4: 输出同步报告

```
## 文档同步报告（增量）

### 变更决策
- [变更-1] §3.2 导出方式: 一次性 → 分页 | 丁昂

### 更新的文档
| 文档 | 版本 | 变更章节 |
|------|------|---------|
| PRD-001.md | 1.1→1.2 | §3.2 |
| SPEC-001-tool-policy-engine.md | 1.0→1.1 | §4.1 |

### 新建的 changelog
- PRD-001-Tool-Policy-Engine.changelog.md（首次创建）
```

## 全量同步模式（`/doc-sync-after-dev`）

### 触发时机

| 场景 | 说明 |
|------|------|
| feature 分支开发完成，准备合并 | CLAUDE.md 规则要求 |
| Code Review 发现文档不一致 | 手动调用 |
| 用户主动说"全量同步"、"doc sync" | 手动调用 |

### Step 1: 收集变更信息

```bash
# 1.1 确认当前分支和 base branch
git branch --show-current
git log --oneline master..HEAD

# 1.2 查看变更的文件列表
git diff master...HEAD --stat

# 1.3 确认身份
git config user.name  # → 推导变更人
```

**操作**：
1. 从分支名推导涉及的 **模块**（domain / infrastructure / application / interfaces）
2. 从变更文件列表推导涉及的 **子模块**
3. 读取当前 sprint 的任务文件：`sprints/active/{最新日期}/*/tasks-{gitid}.md`
4. 从任务文件中提取 **详设引用**（`详设:` 字段指向的文档路径）
5. 定位 **关联需求文档**：
   - 需求：`sprints/backlog/requirements/PRD-*.md`
   - 从详设文档头部 `关联需求` 字段反查
   - 链式追溯：任务文件 → 详设 → 需求

**输出**：变更范围摘要
```
层: backend
模块: tool-policy
子模块: tool system
分支: feature/infra-tool-policy
变更文件: 28 files changed
关联任务: sprints/active/2026-04-01/wave1-xxx/tasks-dingang.md
关联详设:
  - docs/specs/SPEC-001-tool-policy-engine.md
关联需求:
  - sprints/backlog/requirements/PRD-001-Tool-Policy-Engine.md
```

### Step 2: 对比代码与文档

**读取并对比以下维度**：

| 维度 | 代码中的来源 | 文档中的位置 |
|------|-------------|-------------|
| REST API | FastAPI Router 中的端点 | 详设 "API Design" 章节 |
| 数据模型 | Pydantic Model 类 | 详设 "Data Model" 章节 |
| 前端页面 | `frontend/*.html` 文件 | 详设 "组件/页面清单" 章节 |
| 配置项 | `config.yaml` / `.env` | 详设 "Configuration" 章节 |
| 依赖关系 | `requirements.txt` / `pyproject.toml` | 详设 "Dependencies" 章节 |
| 验收标准 | 实际功能行为 | 需求文档 "验收标准" 章节 |
| 功能范围 | 已实现的功能列表 | 需求文档 "功能需求" 章节 |

**识别差异类型**：
- **新增**：代码中有，文档中没有
- **修改**：代码与文档描述不一致
- **废弃**：文档中有，代码中已移除或不实现
- **一致**：无需更新

**输出**：差异清单
```
[新增] BatchExportService — 详设/需求中均未提及
[修改] ExportService.export() 参数新增 pageSize — 需求中未体现分页
[废弃] ExportService.exportAll() — 需求中有，实际未实现（推迟）
[一致] SearchService.search() — 文档与代码一致
```

### Step 3: 更新文档

对每个关联文档（需求 + 详设）按增量模式 Step 3 相同规则执行：

1. 检查并迁移旧格式（规则 7）
2. 递增版本号（规则 6）
3. 更新正文内容
4. 添加正文章节标记（规则 3）
5. 追加 changelog 条目（规则 4）

### Step 4: 清理增量标记

检查文档中是否有之前增量同步留下的标记。全量同步时确认这些标记仍然准确：
- 版本标记是否正确（可能增量时标了 v1.1，全量时需要归并为 v1.2）
- 变更摘要是否准确

### Step 5: 更新任务文档

对 `sprints/active/{日期}/*/tasks-{gitid}.md` 执行：

#### 5.1 标注实际完成情况

```markdown
## 实际完成 vs 原计划

| 原计划任务 | 状态 | 备注 |
|-----------|------|------|
| T1: 实现工具发现 | ✅ 完成 | 与计划一致 |
| T2: 实现导出功能 | ✅ 完成 | 改为分页导出（计划外变更） |
| T3: 实现实时推送 | ⏭️ 推迟 | 推迟到 Wave4，依赖 WebSocket 集成 |
```

#### 5.2 补充计划外新增

```markdown
### 计划外新增
- ✅ [EXTRA-1] 分页导出 — 原一次性导出性能不足，重新设计
- ✅ [EXTRA-2] 批量操作 API — 开发中发现需要批量处理
```

### Step 6: 生成同步报告

```
## 文档同步报告（全量）

### 差异统计
| 类型 | 数量 |
|------|------|
| 新增 | 2 |
| 修改 | 3 |
| 废弃 | 1 |
| 一致 | 12 |

### 更新的文档
| 文档 | 版本 | 变更章节 | 主要变更 |
|------|------|---------|---------|
| PRD-001.md | 1.1→1.2 | §3.2, §5.1 | +分页导出, +批量API, -一次性导出 |
| SPEC-001-tool-policy-engine.md | 1.0→1.1 | §4.1, §6.2 | 同上 |

### 新建/迁移的 changelog
- PRD-001-Tool-Policy-Engine.changelog.md（从旧变更记录迁移）
- SPEC-001-tool-policy-engine.changelog.md（首次创建）

### 任务文档
| 文档 | 完成 | 推迟 | 计划外 |
|------|------|------|--------|
| tasks-dingang.md | 3/4 | 1 | 2 |

### ⚠️ 需要关注
- 跨层影响：修改了 domain/agent/models.py，影响 application/ 和 interfaces/
- 推迟项：实时推送 → 需记入 backlog
```

## 特殊情况处理

### 找不到关联文档

按以下规则定位：
1. **详设**：从模块名查找 `docs/specs/`
2. **需求**：从详设头部 `关联需求` 字段反查，或搜索 `sprints/backlog/requirements/`
3. 如果都找不到，提示用户手动指定

### changelog 文件不存在

直接创建，按规则 4 格式。将当前正文头部的 Version 和 Date 作为 v1.0 初始条目。

### 涉及跨层变更

在同步报告中明确提醒：

```
⚠️ 跨层影响：
- 修改了 domain/agent/models.py
- 影响层: infrastructure/tools（工具注册）、application/（用例层）
- 建议检查相关模块的详设文档是否需要更新
```

### 没有差异

如果代码与文档完全一致，仅更新正文头部 Date 和 Status，不递增版本号。

## Checklist（每次执行完毕前确认）

### 增量模式 checklist
- [ ] 变更决策已从对话中完整提取
- [ ] 所有关联文档已定位（需求 + 详设，双向）
- [ ] 旧格式变更记录已迁移（如果是首次对该文档执行）
- [ ] 正文版本号和日期已递增
- [ ] 正文内容已更新为当前最新状态（废弃的已删除）
- [ ] 变更章节已加版本标记和摘要行
- [ ] changelog 条目已追加（含章节号、类型、原因、关联文档）
- [ ] 同步报告已输出

### 全量模式 checklist（包含以上全部，另加）
