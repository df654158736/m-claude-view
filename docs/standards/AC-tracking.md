# AC（需求点）拆分与追踪规范

> **适用范围**：所有 PRD 的"需求点/验收标准"章节，以及引用 AC 的任务文档
> **核心原则**：AC 是**验收契约**。PRD 创建阶段拆，开发阶段冻结，变更走 changelog。
> **机读**：AC ID 格式、状态标识、Task 引用语法被 `/prd-status` skill 强制解析

---

## 1. AC 是什么

**AC (Acceptance Criteria) = 需求点 = 验收单位**

一个 AC 是业务方、PM、Dev、QA 四方达成共识的**最小验收单元**，具备以下特性：

1. **独立可验收** — 不依赖其他 AC 即可验证通过 / 失败
2. **独立可开发** — 一个 AC 对应 1~N 个 Task，不会跨 AC 共用 Task
3. **业务可理解** — PM 和业务方都能读懂，不含实现细节
4. **粒度适中** — 典型估算 0.5 ~ 3 人日；超过 3 人日要考虑拆更细

## 2. AC ID 规范

### 2.1 格式

```
AC-{PRD编号}-{两位序号}
```

示例：
- `AC-002D-01` — PRD-002D 的第 1 个 AC
- `AC-008-05` — PRD-008 的第 5 个 AC
- `AC-030-12` — PRD-030 的第 12 个 AC

### 2.2 正则

```regex
AC-\d{3}[A-Z]?-\d{2}
```

### 2.3 命名约束

| 规则 | 说明 |
|------|------|
| 序号从 01 开始 | 单位数用前导 0：`AC-002D-01` 不是 `AC-002D-1` |
| 序号不复用 | 即使某 AC 被废弃，它的编号不能被新 AC 占用 |
| 序号在 PRD 内唯一 | 不同 PRD 可有相同序号（被 PRD 编号区分） |
| PRD 合并/拆分时保留旧 ID | 老 AC 保留原编号，新 AC 继续往下排 |

## 3. PRD 内的 AC 章节格式

### 3.1 标准格式

在 PRD 正文中，AC 章节的写法：

```markdown
## 需求点（可追踪）

### AC-002D-01 支持 ≥3 层对象类型嵌套  `P0`

> **状态**: 开发✅ · 验收已验收
> **负责**: dev=丁昂 · qa=self-test
> **详设**: SPEC-BE-014 §3.1
> **预估**: 2 人日
> **验收通过**: 2026-04-10

**描述**:
系统必须支持创建层级深度 ≥3 层的对象类型嵌套结构，UI 能正确展示树形，查询性能 P95 < 200ms。

**BDD 验收**:
```gherkin
Scenario: 创建 3 层嵌套对象类型
  Given 已有顶层对象类型"组织"
  When 用户创建子类型"部门"，再创建孙类型"团队"
  Then 对象管理器显示三级树形结构
  And 穿透查询 Organization → Department → Team 成功
```

---

### AC-002D-02 父子关系反向查询  `P0`

> **状态**: 开发🟡 · 验收未验收
> **负责**: dev=丁昂 · qa=self-test
> **详设**: SPEC-BE-014 §3.2
> **依赖**: AC-002D-01
> **预估**: 1.5 人日

**描述**:
从任意子对象实例可以反向查询到所有祖先对象实例，返回完整路径。

**BDD 验收**:
（省略）
```

### 3.2 字段说明

**AC 级字段**（人工维护 + 自动回写混合）：

| 字段 | 类型 | 维护方 | 说明 |
|------|------|-------|------|
| ID 与标题 | 必填 | PM | `### AC-002D-01 标题  \`P0\`` |
| 优先级 | 必填 | PM | `P0` / `P1` / `P2`，写在标题反引号内 |
| 状态 | 自动 | `/prd-status update` | `开发⬜/🟡/🟢/✅` · `验收未验收/已验收/未通过` |
| 负责 | 必填 | PM | `dev=xxx · qa=yyy`（当前 qa 固定写 `self-test`） |
| 详设 | 必填 | PM | SPEC 编号 + 章节号（如 `SPEC-BE-014 §3.1`） |
| 关联任务 | 自动 | skill 扫描 | task 文件路径 + checkbox ID（如 `tasks-丁昂.md#T1`） |
| 关联代码 | 自动 | skill 扫描 | 实现代码主路径（前端 `.vue` / 后端 Controller / Proto）；多个用逗号分隔 |
| 依赖 | 可选 | PM | 跨 PRD 或本 PRD 内的前置 AC |
| 预估 | 可选 | PM | 人日 |
| 实际 | 自动 | skill | 关联 task 累加 |
| 自测通过 | 自动 | `/qg` 或 `/prd-status mark-tested` | 日期 + 触发人 + 证据（qg report 链接 / commit SHA） |
| 验收通过 | 自动 | `/prd-status accept` | 日期 + PM |
| 描述 | 必填 | PM | 业务语言描述 |
| BDD 验收 | 必填 | PM | Given/When/Then 格式 |

### 3.3 AC 标题规范（强制）

AC 标题要让**没读过 PRD 的业务方**也能一眼看懂在讲什么功能。

**硬性要求**：

| 要求 | 说明 |
|------|------|
| 字数 | **15 ~ 40 个中文字符**（宽松上限 60） |
| 结构 | `动词 + 对象 + 场景/约束`（三段式） |
| 自洽 | 不依赖上下文（如"之前那个"、"上面说的"）也能理解 |
| 含语义 | 不只是功能代号（如纯"周边搜索" → 要说清搜什么） |

**❌ 反模式（字数不足/过于抽象）**：

| 坏标题 | 问题 |
|-------|------|
| `周边搜索` | 搜什么周边？节点？实体？时间段？ |
| `步骤卡片独立编辑` | 什么步骤？编辑什么？为什么独立？ |
| `Header 双 Tab 切换器` | 切换什么？解决什么问题？ |
| `Agent 编排` | 编排什么？可视化？代码？ |
| `关系筛选` | 筛选哪种关系？什么维度？ |

**✅ 好标题（含三段式）**：

| 好标题 | 动词 | 对象 | 场景/约束 |
|-------|------|------|----------|
| `以节点为中心展开周边关系（可配深度/类型）` | 展开 | 周边关系 | 以节点为中心 + 可配深度/类型 |
| `Pipeline 步骤卡片支持独立编辑并实时同步下游` | 编辑+同步 | Pipeline 步骤卡片 | 独立 + 实时下游 |
| `Header 双 Tab 在业务端/实施端之间切换工作台` | 切换 | 工作台 | Header 双 Tab + 业务端/实施端 |
| `可视化拖拽编排 AI Agent 工作流（多节点组合）` | 编排 | AI Agent 工作流 | 可视化拖拽 + 多节点组合 |
| `按类型/方向/时间/属性值筛选图谱中的关系边` | 筛选 | 图谱关系边 | 按类型/方向/时间/属性值 |

### 3.4 描述字段规范

标题写的是"做什么"，**描述字段**展开"为什么 + 怎么做的约束"：

- 1~3 句话
- 说清业务价值（为什么用户需要这个功能）
- 说清关键约束（如性能指标、数据范围、权限边界）
- 禁止以下内容：

### 3.5 不允许在 AC 描述里出现的内容

- ❌ 实现细节（"用 PostgreSQL 的 CTE 做递归查询"）
- ❌ 技术栈选型（"用 Vue 3 + Element Plus"）
- ❌ 模糊形容词（"快速响应"、"用户友好"、"流畅" — 必须改成可量化指标）
- ❌ 未定义的代称（"上次那个"、"之前讨论的"）

## 4. Task 引用 AC 的规范

### 4.1 任务文档头部

```markdown
# Tasks — 丁昂 — 2026-04-16 Wave3

> **Sprint**: 2026-04-16
> **Focus**: 多层级建模前端实现
> **分支**: feature/app-ontology-manager-multihierarchy
>
> **关联需求点**:
>   - AC-002D-01 支持 ≥3 层对象类型嵌套
>   - AC-002D-02 父子关系反向查询
>   - AC-002D-03 多层级下钻 UI
>
> **关联详设**: SPEC-BE-014 §3.1-3.3, SPEC-FE-005B §4
```

### 4.2 任务级引用（checkbox 级）

一个 task 文件可能覆盖多个 AC，用 checkbox 级引用精确定位：

```markdown
## Tasks

### T1: 后端多层级递归查询

- [x] 实现 ParentChildRepository (AC-002D-01)
- [x] 新增 /api/ontology/drilldown 接口 (AC-002D-02)
- [ ] 单元测试覆盖 (AC-002D-01, AC-002D-02)

### T2: 前端树形组件

- [ ] 实现 tool-list 页面组件 (AC-002D-03)
- [ ] 接入工具详情路由 (AC-002D-03)
- [ ] E2E 测试 (AC-002D-03)
```

### 4.3 引用语法规则

| 规则 | 说明 |
|------|------|
| AC ID 用英文圆括号包裹 | `(AC-002D-01)` |
| 多 AC 用英文逗号 + 空格 | `(AC-002D-01, AC-002D-02)` |
| 放在 checkbox 行末 | 便于正则扫描 |
| 不用方括号 | 方括号是 markdown checkbox 语法，避免冲突 |

### 4.4 聚合规则

`/prd-status update` 扫 task 文件时：

```
对每个 AC：
  - 扫所有 tasks-*.md 文件
  - 找到所有 "(AC-XXX)" 引用的 checkbox
  - 统计 [x] / 全部 比例
  - 按 §3 聚合规则映射到 AC 状态
```

## 5. AC 拆分时机

### 5.1 拆分时机矩阵

| PRD 状态 | AC 要求 | 拆分方式 |
|---------|--------|---------|
| `draft` 草稿 | AC 可粗，只写核心功能点 | `/PRD` skill 引导 |
| `reviewed` 已评审 | **AC 必须完整**，每个 AC 可独立验收 | `/PRD` skill 强制检查 |
| `in-dev` 开发中 | **AC 冻结**，任何新增/修改走变更流程 | `/doc-sync` 触发 changelog |
| `delivered` 已交付 | AC 全部 ✅ | 只读 |

### 5.2 已有 PRD 补 AC

对于**已经写好但缺 AC** 或 **AC 不完整** 的历史 PRD：

1. PM 调用 `/PRD add-ac PRD-002D`
2. Skill 扫描 PRD 正文，识别潜在的需求点（从"功能需求"、"验收标准"、"功能列表"等章节）
3. 生成 AC 草稿（每条附带 BDD 骨架）
4. PM 逐条评审、调整、确认
5. 按格式写入 PRD 的"## 需求点"章节
6. 自动触发 `/prd-status update` 刷新 PRD-STATUS.md

## 6. 状态机

### 6.1 AC 状态

```
⬜ 未开始
   │ 首次关联 task 且 task 开始勾 [x]
   ▼
🟡 开发中
   │ 关联的所有 task checkbox 全 [x]
   ▼
🟢 已自测 （等 PM 验收）
   │                    │
   │ PM 验收通过         │ PM 验收不通过
   ▼                    ▼
✅ 已验收             🔴 验收未通过
                        │ Dev 修复后重新自测
                        └──→ 回到 🟡
```

### 6.2 验收字段

`/prd-status accept AC-002D-01` 会在 PRD 内该 AC 的元数据块追加：

```markdown
> **状态**: 开发✅ · 验收已验收
> **验收通过**: 2026-04-16 丁昂
```

`/prd-status reject AC-002D-01 "BDD 第三条场景未覆盖"` 会追加：

```markdown
> **状态**: 开发✅ · 验收未通过
> **验收退回**: 2026-04-16 丁昂 · 理由：BDD 第三条场景未覆盖
```

同时在 `.changelog.md` 追加条目。

## 7. AC 变更流程（`reviewed` 或之后的状态）

AC 进入 `reviewed` 后冻结，任何变更都要走：

```
PM 想改/加/删 AC
    ↓
在 PRD 正文改 AC 章节（直接改或追加）
    ↓
触发 /doc-sync（自动或手动）
    ↓
/doc-sync 检测到 AC 变更，追加 .changelog.md 条目
    ↓
/doc-sync 末尾自动调 /prd-status update
    ↓
PRD-STATUS.md 刷新 · "最近变更"段追加一行
```

### 7.1 .changelog.md 中 AC 变更的条目格式

```markdown
## [v1.2] 2026-04-16 | 丁昂

- **AC-002D-03** 新增 → 多层级下钻 UI | 原因: 用户评审发现缺少 UI 验收标准 | 关联: SPEC-FE-005B §4 需补充
- **AC-002D-01** 修改 → 性能指标从 "P95 < 500ms" 改为 "P95 < 200ms"
  - 原内容: 查询性能 P95 < 500ms
  - 现内容: 查询性能 P95 < 200ms
  - 原因: 实测发现 500ms 太宽松，用户可感知延迟
- **AC-002D-08** 废弃 → 推迟到 v2026-05-30 | 原因: 本版本资源紧张 | 替代: 无
```

## 8. 质量检查（`/prd-status check`）

| 检查项 | 严重性 |
|--------|-------|
| AC ID 不符合 `AC-\d{3}[A-Z]?-\d{2}` 格式 | 🔴 |
| AC 标题反引号内的优先级非 P0/P1/P2 | 🔴 |
| PRD 状态 `in-dev` 但有 AC 无关联 task | 🟡 |
| AC 状态 ✅ 但关联 task 还有 `[ ]` | 🔴 |
| AC 无 BDD 验收段（状态 >= reviewed） | 🟡 |
| AC 描述中出现模糊形容词（快速/流畅/友好/方便）| 🟡 |
| Task 引用了不存在的 AC ID | 🔴 |
| Task 未引用任何 AC（孤儿任务） | 🔴 |
| 同 PRD 内 AC 序号重复 | 🔴 |

## 9. 速查表（给 PM 贴墙）

**写 AC 的 6 个问题**：

1. 这条 AC 能独立验收通过/失败吗？
2. 业务方看得懂吗？不含实现细节？
3. 有 BDD 三段式吗？
4. 验收指标可量化吗（没有"快速"、"流畅"）？
5. 优先级标了吗（P0/P1/P2）？
6. 有没有依赖其他 AC？标出来了吗？

**Dev 勾 checkbox 的 3 个问题**：

1. 这个 checkbox 引用了哪个 AC？`(AC-XXX-YY)` 写了吗？
2. 这个 AC 的所有 checkbox 都 `[x]` 了吗？→ Claude 会自动告知 PM 可验收
3. 如果发现 AC 描述不准，是不是该走 `/doc-sync`？

## 10. 自测 Definition of Done（研发自测档 DoD）

> 一个 AC 从 🟡 开发中 → 🟢 已自测 的**硬性条件**。六条全达才能标自测通过，任何一条未达保持 🟡。

### 10.1 六条硬条件

| # | 条件 | 验证方式 | 备注 |
|---|------|---------|------|
| 1 | **单元测试通过** | `PYTHONPATH=backend pytest backend/tests/ -v` 全绿 | 新增代码 ≥90% 覆盖，核心类 ≥90%，整体 ≥80% |
| 2 | **契约/集成测试通过** | 集成测试 + 涉及外部系统加 Mock 测试 | 跨层调用必须有集成测试 |
| 3 | **/qg 绿灯** | `/qg <module>` 跑 API/E2E/场景测试 | 该 AC 所属模块的场景测试必须绿 |
| 4 | **主流程手动验证 ≥1 次** | dev 本地跑通一遍 | UI 类 AC 必须，API 类可跳过 |
| 5 | **代码已合并到主干** | commit SHA 可追溯 | PR 已合并 or 直接 push master |
| 6 | **无 console error / warning** | 浏览器控制台 / 服务端日志 clean | 业务影响的 warning 必须消除 |

### 10.2 录入方式（推荐组合）

| 方式 | 触发方 | 时机 | 如何记录 |
|------|-------|------|---------|
| **A. `/qg` 自动** | Dev 跑 `/qg <module>` | 测试绿灯后自动 | `/qg` skill 调 `/prd-status mark-tested`，写入 PRD AC 章节 |
| **B. 手动标记** | Dev 对话中声明 | 自测完成后 | Claude 调 `/prd-status mark-tested AC-XXX "证据"` |
| **C. Task 驱动** | Dev 勾 task checkbox | commit 时 | task 文件 `- [x] 实现 XX (AC-002D-01) [自测 2026-04-17]` |

**推荐组合**：A 为主（自动化） + B 兜底（UI 细节） + C 增强（task 天然锚定）。

### 10.3 自测通过后 AC 元数据块的写入格式

Claude 自动在 PRD 的 AC 章节追加：

```markdown
> **自测通过**: 2026-04-17 丁昂 · qg-report-1713334567 · commit@abc1234
```

三个组成部分：
- **日期 + 触发人** — 谁自测的（git id）
- **证据 1：`/qg` 报告 ID** — `qg-report-<timestamp>`（若是方式 A）
- **证据 2：commit SHA** — 自测时的代码状态（短 SHA）

### 10.4 验收环节（PM 档）

- Dev 标"自测通过"后，Claude 主动告知 PM：**"AC-XXX 已自测（证据：...），要验收吗？"**
- PM 说"验收通过 AC-XXX" → Claude 调 `/prd-status accept AC-XXX`
- **硬约束**：`/prd-status accept` 会先检查"自测通过"字段是否已填，**未填则阻断**（不允许跳过自测）
- 写入格式：`> **验收通过**: 2026-04-17 丁昂`

### 10.5 反模式（禁止）

- ❌ 跳过自测直接让 PM 验收（`/prd-status accept` 会阻断）
- ❌ `@Ignore` 跳过失败测试来凑绿灯
- ❌ 手动改"自测通过"字段造假（skill 校验 commit SHA 与字段日期）
- ❌ DoD 部分达标就标"已自测"（必须 6 条全部过）
